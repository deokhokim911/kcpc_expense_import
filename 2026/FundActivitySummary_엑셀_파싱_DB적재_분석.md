# Fund Activity 엑셀 → DB 적재 분석

**목적**: `2026/migration.sql`로 정의된 `fund_activity_import_run`, `fund_activity_summary_row`에 대해, 실제 원본 엑셀(`FundActivitySummary.20260503.xlsx`, `GeneralLedger.20260501.Special_web_posting.xlsx`) 구조와 참조 문서(`FundActivitySummary_년도별특별_UI_및_단계별_개발.md`)를 바탕으로 **컬럼 매핑·적재 순서·파싱 시 유의사항**을 정리한다.

---

## 구현·실행 상태 (갱신: 2026-05-07)

| 항목 | 상태 |
|------|------|
| **`public.org_*` 테이블** | **적재 로직에서 기본적으로 호출하지 않음.** 원장 → `org_balance` 삭제·INSERT 경로는 `-import-org-balance`를 줄 때만 실행된다. |
| **`fund_activity_summary_row` 적재** | **구현 완료.** 리포지토리 루트에서 `fund_activity_import.go` + `main.go` 플래그로 동작. 한 트랜잭션에서 `fund_activity_import_run` 1행 생성 후 요약 행 일괄 INSERT. |
| **`fund_activity_import_run`** | Summary 적재 시 **필수**(FK). 기간은 Summary 시트 제목 행에서 파싱. 선택적으로 `-detail-xlsx`로 G/L 파일 기간 일치 검사 및 `detail_source_filename` / `detail_imported_at` 설정. |
| **샘플 dry-run** | `FundActivitySummary.20260503.xlsx` + `GeneralLedger.20260501.Special_web_posting.xlsx` 조합으로 **31행** 요약 + Total 포함 적재 예정으로 확인됨. |

**실행 예**

```bash
# 파싱만 (DB 없음)
go run . -import-fund-summary -summary-xlsx ./2026/FundActivitySummary.20260503.xlsx -dry-run

# 기간 교차 검증 포함 dry-run
go run . -import-fund-summary -summary-xlsx ./2026/FundActivitySummary.20260503.xlsx \
  -detail-xlsx ./2026/GeneralLedger.20260501.Special_web_posting.xlsx -dry-run

# 실제 적재 (DB 연결·마이그레이션 선행 필요)
go run . -import-fund-summary -summary-xlsx ./2026/FundActivitySummary.20260503.xlsx \
  -fiscal-year 2026
```

**레거시**: `-import-org-balance` 없이 바이너리만 실행하면 아무 DB 작업도 하지 않고 안내 메시지 후 종료한다.

## 1. 테이블 역할과 파일 매핑 (관계 정리)

| 파일 | DB 테이블 | 비고 |
|------|-----------|------|
| `FundActivitySummary.*.xlsx` | **`fund_activity_summary_row`** | 펀드별 요약 행 (여러 INSERT) |
| 두 파일 공통 | **`fund_activity_import_run`** | **한 번의 import 단위당 1행** (메타데이터: 기간, 회계연도, 파일명, 적재 시각) |

**중요**: `fund_activity_import_run`에는 금액·펀드명·원장 라인이 **없다**. 원장 상세는 참조 문서의 Phase 2에서 도입 예정인 **`fund_activity_detail_line`(가칭)** 에 넣는 것이 맞다.  
따라서 **General Ledger 엑셀 전체를 “한 줄씩 `fund_activity_import_run`에 넣는다”는 것은 스키마상 불가능**하며, G/L 파일에서 이 테이블로 가져올 수 있는 것은 **파일명·적재 시각·기간 문자열 검증** 등 **실행(run) 메타데이터**뿐이다.

실무적인 적재 순서는 다음과 같다.

1. 두 파일에서 **동일한 보고 기간**(`period_start`, `period_end`)을 추출해 일치하는지 검증한다.
2. `fund_activity_import_run`에 **1행 INSERT** → 생성된 `id`를 `run_id`로 사용한다.
3. Summary 시트를 파싱해 **`fund_activity_summary_row`에 다건 INSERT** (`run_id` FK).
4. (향후) G/L 시트를 파싱해 **`fund_activity_detail_line`** 에 다건 INSERT — 현재 마이그레이션 범위에는 해당 테이블이 없음.

## 2. 원본 엑셀 구조 (실측: `FundActivitySummary.20260503.xlsx`)

- **시트 이름**: `Fund Activity Summary` (참조 문서와 동일)
- **상단 고정**: 행 6까지 헤더·타이틀 영역, 데이터는 **행 7부터** (`pane` 기준 `topLeftCell="A7"`)

### 2.1 기간·생성 정보 (메타)

| 행 | 내용 예시 | 용도 |
|----|-----------|------|
| 2 | `Fund Activity Summary for the period of 10/01/2025 to 05/01/2026` | `period_start`, `period_end` 파싱 |
| 4 | `Generated by … on 2026-05-03 11:43` | (선택) `notes` 또는 감사 로그 |

**기간 파싱 예**: 문구에서 `MM/DD/YYYY` 두 개를 추출 → `period_start = 2025-10-01`, `period_end = 2026-05-01` (로컬 DATE로 저장).

### 2.2 헤더 행 (행 6)

엑셀 열과 DB 컬럼 대응은 아래와 같다.

| 열 | 엑셀 헤더 | DB (`fund_activity_summary_row`) |
|----|-----------|-----------------------------------|
| A | Fund | `fund_name` |
| B | Beginning Balance | `beginning_balance` |
| C | Income | `income` |
| D | Expenses | `expenses` |
| E | Net Income (Expense) | `net_income_expense` |
| F | Transfer | `transfer` |
| G | Net Increase (Decrease) | `net_increase_decrease` |
| H | Ending Balance | `ending_balance` |
| I | [Beginning of Fiscal Year] Balance | `beginning_of_fiscal_year_balance` |

### 2.3 데이터 행·Total 행

- **행 7 ~ (Total 직전)**: 일반 펀드 행 → `is_total_row = false`
- **마지막 요약 행**: 실측 파일에서 **행 37**, 열 A 문자열이 **`Total`** → `is_total_row = true`

파서는 헤더 다음부터 순회하면서:

- 열 A가 비어 있으면 스킵
- `strings.TrimSpace` 후 `strings.EqualFold(colA, "Total")` 이면 Total 행으로 처리

숫자는 엑셀에서 이미 숫자 타입으로 저장되어 있으므로, 라이브러리에서 `float64`/`decimal`으로 읽어 `DECIMAL(20,6)`에 맞게 넣으면 된다.

### 2.4 `fiscalyear` (import_run)

Summary 파일만으로 FY 라벨이 한 컬럼에 박혀 있지 않다. 운영 규칙 예시:

- UI·조회에서 **회계연도 2026**을 쓰는 경우(참조 앱과 동일): 상수 또는 설정으로 `fiscalyear = 2026`
- 또는 `period_end`의 연도/월로 회계연도를 도출하는 규칙을 별도 문서로 고정

## 3. 원본 엑셀 구조 (실측: `GeneralLedger.20260501.Special_web_posting.xlsx`)

- **시트 이름**: `General Ledger`

### 3.1 기간·생성 정보

| 행 | 내용 예시 |
|----|-----------|
| 2 | `General Ledger for the period of 10/01/2025 to 05/01/2026` |
| 4 | `Generated by … on 2026-05-01 15:39` |

Summary 파일의 기간 문자열과 **동일해야** 같은 `run`으로 묶을 수 있다.

### 3.2 헤더 행 (실측 행 6)

`Name, Date, #, Type, Contact, Memo, Reference Number, Note, Fund, Ministries, Sub-Ministries, STM Tags, Debit, Credit, Amount, Balance`

이후 행은 QB 스타일로 **계정 구간 헤더**(예: `Asset`, `1865.1 - Contractors`), **Beginning Balance**, 거래 라인, **Total for …** 등이 섞여 있다.

### 3.3 `fund_activity_import_run`에 넣을 수 있는 것 (G/L 기준)

| 컬럼 | 소스 |
|------|------|
| `period_start`, `period_end` | 행 2 문구 파싱 (Summary와 교차 검증) |
| `detail_source_filename` | 파일명 예: `GeneralLedger.20260501.Special_web_posting.xlsx` |
| `detail_imported_at` | 적재 실행 시각 (`CURRENT_TIMESTAMP` 또는 배치 시작 시각) |
| `notes` | (선택) 행 4 생성 정보 문자열 |

G/L **거래 라인 전체**는 이 테이블에 들어가지 않는다.

## 4. 참조 문서와의 정합성

`FundActivitySummary_년도별특별_UI_및_단계별_개발.md` 요지:

- Summary는 **`Fund Activity Summary` 시트**, Detail은 **`General Ledger` 시트** — 실측 파일과 일치
- Import 단위는 **`import_run`(현재 테이블명 `fund_activity_import_run`)** 에 Summary·Detail을 묶음
- Summary 행과 Detail 행은 **`run_id` + `fund_name`(Fund 열, 문자열 트림)** 으로 연결
- Income / Expenses 드릴다운은 Detail에서 Credit·Debit 규칙 적용 — **Detail 적재 테이블이 선행**되어야 함 (문서 Phase 2)

현재 `migration.sql`에는 Summary까지만 있으므로, **G/L 라인 적재는 다음 마이그레이션·파서 단계**로 분리하는 것이 설계와 일치한다.

## 5. 권장 적재 알고리즘 (의사코드)

```
parse_period(summary_row2_text, gl_row2_text) -> (start, end)
  assert_equal(parsed(summary), parsed(gl)) or fail

run_id = INSERT fund_activity_import_run (
  fiscalyear := configured FY,
  period_start, period_end,
  summary_imported_at := now(),
  detail_imported_at := now(),           -- G/L도 같은 실행에서 처리했다면
  summary_source_filename := "...FundActivitySummary.20260503.xlsx",
  detail_source_filename := "...Special_web_posting.xlsx",
  notes := optional
) RETURNING id

FOR each data row in sheet "Fund Activity Summary" AFTER header row:
  IF col A is Total:
    INSERT fund_activity_summary_row (run_id, ..., is_total_row=true)
  ELSE:
    INSERT fund_activity_summary_row (run_id, ..., is_total_row=false)

-- 향후 Phase:
FOR each logical transaction line in "General Ledger":
  INSERT fund_activity_detail_line (run_id, fund_name from col Fund, ...)
```

## 6. 구현 시 기술 유의사항

1. **인라인 문자열**: Summary 파일은 `Fund` 등에 `inlineStr`이 사용된다. **excelize** 등으로 읽으면 처리되지만, 단순 XML 파서만 쓸 경우 sharedStrings가 아닌 인라인 문자열을 반드시 처리해야 한다.
2. **헤더 행 위치**: 현재 샘플은 행 6이 헤더이지만, QB 내보내기 버전에 따라 한 줄 어긋날 수 있으므로 **`Fund`·`Beginning Balance` 텍스트로 헤더 행을 탐색**하는 편이 안전하다.
3. **부동소수**: 금액은 DB에서 `NUMERIC`/`DECIMAL`로 저장하고, 애플리케이션에서는 문자열로 받은 숫자에 대해 콤마 제거 후 파싱한다.
4. **기존 `main.go`**: `GeneralLedger.*.Special_web_posting.xlsx`는 이미 `loadWorksheetRows`로 읽어 **`org_balance`** 적재용으로 파싱한다. 새 테이블 적재는 **별도 파서·별도 INSERT 경로**로 두는 것이 migration.sql 주석(“적재는 앱이 아닌 별도 파서”)과 맞다.
